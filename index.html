Le killcam — tout le monde connaît

Vous mourez dans Counter-Strike, Overwatch, Fortnite…

Le jeu vous montre les 5 dernières secondes depuis le point de vue de votre adversaire.

Question : le jeu a-t-il enregistré une vidéo de ces 5 secondes en permanence ?

Non. Jamais.

Une vidéo 60fps 1080p = ~150 Mo/min.
Une session de jeu de 30 min = 4,5 Go.
En permanence, en RAM. Non.

Ce que le jeu stocke vraiment

[t=1] MOVE_RIGHT p=1
[t=4] JUMP p=1

La formule du replay

État initial + Log d'events = Replay identique
État initial

Position de départ, graine du générateur aléatoire (RNG seed), configuration de la map, stats des joueurs.

→ Reproductible à l'identique

Log d'events

Uniquement les changements d'inputs (pas 60 snapshots/sec), plus les événements système (mort, victoire).

→ Append-only, compact

Résultat

En réinjectant les events dans le moteur physique depuis l'état initial, on reconstruit exactement la même partie.

→ Déterministe

Le moteur physique est une fonction pure : f(état, input) → nouvel_état
RALENTIR — c'est la slide pivot de toute la partie 1. Tout ce qui suit en découle. Prendre le temps de la formule.
LIRE LA FORMULE À VOIX HAUTE, LENTEMENT "État initial... plus log d'events... égale replay identique." Silence d'une seconde. Laisser atterrir. "C'est tout. C'est le principe entier."
État initial :
"C'est le point de départ. Dans notre mini-jeu : la position du joueur, la seed qui a généré les plateformes, la gravité. Rien d'autre. Ce n'est pas une capture complète de la partie — c'est juste ce dont le moteur a besoin pour partir de zéro." Insister : la seed RNG est critique. Si deux machines partent avec la même seed, elles génèrent exactement le même niveau. C'est pour ça que le replay est possible sur n'importe quelle machine.
Log d'events :
"On n'enregistre pas 60 photos par seconde. On enregistre uniquement les moments où quelque chose change : le joueur appuie sur droite au tick 4, il saute au tick 9, il lâche droite au tick 23. Entre ces moments, le moteur calcule tout seul. Le log est donc très compact."
Résultat — le déterminisme :
"Si je pars du même état initial et que je rejoue les mêmes events dans le même ordre, j'arrive exactement au même endroit. Toujours. C'est ça, le déterminisme. Ce n'est pas une propriété magique — c'est une contrainte qu'on impose au moteur."
LA HIGHLIGHT-BOX — FONCTION PURE
"f(état, input) → nouvel_état. Même entrée, même sortie. Si ma fonction regarde l'heure système, génère un seed random, fait un appel réseau — deux exécutions identiques peuvent donner des résultats différents. Le replay diverge. C'est pour ça que les jeux fixent leur générateur de hasard avec une seed : pour que 'aléatoire' signifie quand même 'reproductible'. Ce n'est pas un détail technique — c'est un choix d'architecture qu'on fait dès le départ, sinon le replay est IMPOSSIBLE."
TRANSITION VERS LA SUITE
Planter la graine : "Cette formule, on va la retrouver ailleurs. Gardez-la en tête."

Killcam = time travel
STARCRAFT / STRATÉGIE Le fichier .rep ne contient pas les positions de toutes les unités à chaque instant — ça ferait des gigaoctets. Il contient uniquement les ordres des joueurs : "à 2 minutes 14, le joueur 1 a dit à cette unité d'aller là". C'est tout. Le jeu recalcule le reste tout seul. Résultat : une heure de jeu avec des centaines d'unités tient dans un fichier plus petit qu'une photo de téléphone. Le revers : si Blizzard sort un patch qui change le comportement d'une unité, les anciens replays deviennent faux — le moteur recalcule différemment. C'est un avant-goût d'un problème qu'on verra en partie 4 avec le versioning des events. Glisser : "même idée, même piège."
DOOM (1993) — LES .LMP Doom tourne à 35 images par seconde. À chaque image, le moteur note en quatre chiffres ce que le joueur est en train de faire : est-ce qu'il avance, est-ce qu'il tourne, est-ce qu'il tire. C'est tout. Quatre chiffres, 35 fois par seconde. La démo qu'on voit tourner au lancement du jeu — le marine qui se balade dans le premier niveau — c'est juste ce fichier qui se rejoue. Quelques Ko. Pas une vidéo. Les speedrunners l'utilisent encore aujourd'hui pour soumettre leurs records : ils envoient le fichier .lmp, pas une capture d'écran. Le fichier est la preuve parce que n'importe qui peut le rejouer et vérifier. Marteler : 1993. Cette idée a 32 ans. Ce n'est pas une tendance architecturale récente.
CS2 / FPS MODERNES Quand vous mourez dans CS2 et que le killcam se lance, vous pensez regarder une caméra braquée sur votre tueur pendant les 5 dernières secondes. Cette caméra n'a jamais existé. Ce que le serveur fait en réalité : il a gardé en mémoire tout ce qui s'est passé sur ces 5 secondes — positions, mouvements, actions de chaque joueur. Quand vous mourez, votre client rejoue ces données depuis le point de vue de votre adversaire. C'est une reconstruction, pas un enregistrement. C'est le moment fort pour la salle : tout le monde a vu un killcam, personne ne savait que c'était du rejoue d'events en direct.

Les domain events — immuables, nommés au passé

Règles des events
  • Nommés au passé
    RoomOpened ≠ OpenRoom
  • Immuables — jamais modifiés
  • Append-only — jamais supprimés
  • Portent leur payload complet
  • Horodatés — occurredAt
  • Versionnés dans le stream
// events.ts — types immuables, nommés au passé
  export type DomainEvent =
    | RoomOpened        // ← passé, pas "OpenRoom"
    | AttendeeCheckedIn
    | AttendeeCheckedOut
    | RoomFull          // ← émis AUTO par la règle métier
    | RoomClosed;

  interface BaseEvent {
    eventId:     string;   // identifiant unique
    aggregateId: string;   // roomId — "amphi-a"
    occurredAt:  string;   // ISO 8601 — quand ?
    version:     number;   // position dans le stream
  }

  export interface AttendeeCheckedIn extends BaseEvent {
    type: "AttendeeCheckedIn";
    payload: {
      attendeeId:   string; // qui ?
      attendeeName: string;
    };
  }

LE NOMMAGE AU PASSÉ
"Première règle, et c'est plus qu'une convention : un event se nomme au passé. RoomOpened, pas OpenRoom. AttendeeCheckedIn, pas CheckInAttendee."
"La différence n'est pas stylistique. OpenRoom, c'est un ordre — quelqu'un demande quelque chose. RoomOpened, c'est un fait — quelque chose s'est passé. Un fait ne peut pas être annulé. Il peut seulement être compensé par un autre fait."
"Si j'ai écrit RoomOpened dans mon log, je ne peux pas l'effacer. Si j'ai fait une erreur, j'émets RoomClosed. L'erreur reste visible. La correction aussi."

"Et Immuable ça signifie qu'une fois écrit, un event ne change jamais. Pas de UPDATE sur un event. Jamais. C'est la même contrainte que dans un fichier replay de jeu : on n'efface pas un input du passé. On ajoute la suite."

BASE-EVENT
Montrer les quatre champs de BaseEvent un par un : "eventId : chaque event est unique, identifiable, traçable. aggregateId : sur quel objet métier ça s'est passé — ici la salle. occurredAt : quand. Pas quand on l'a traité — quand ça s'est passé. version : la position dans le stream. C'est l'équivalent du tick dans le jeu. C'est ce qui garantit l'ordre et permet le time travel."

ROOMFULL — LE POINT D'OR
"Concernant RoomFull, il porte un commentaire : émis AUTO par la règle métier. Ce n'est pas le client qui envoie cet event. C'est le domaine qui le produit automatiquement quand la capacité est atteinte."
"C'est là qu'on voit la différence entre une simple liste de logs et un vrai modèle événementiel. Les events ne sont pas des traces de ce que l'utilisateur a fait. Ce sont des faits métier — certains viennent des actions utilisateur, d'autres viennent des règles du domaine lui-même."
Analogie jeu : "Le joueur appuie sur SHOOT. C'est son action. PLAYER_HIT est émis par le moteur si la hitbox correspond. Le joueur ne décide pas si ça touche — le moteur décide."

TRANSITION : "Ces events vont quelque part. C'est la prochaine slide."

L'event store — append-only, toujours

// eventStore.ts — le cœur du système
  class EventStore {
    // Le log global — ici en mémoire pour la démo
    // En prod : PostgreSQL, EventStoreDB, Kafka…
    private events: DomainEvent[] = [];

    // APPEND ONLY — jamais de UPDATE ni DELETE
    append(events: DomainEvent[]): void {
      for (const event of events) {
        this.events.push(event);

        // Log formaté — le parallèle avec le jeu
        const v = String(event.version).padStart(3,' ');
        const type = event.type.padEnd(24,' ');
        const data = JSON.stringify(event.payload);
        console.log(`[${v}] ${type} ${data}`);
      }
    }

    // Tous les events d'une salle, dans l'ordre
    getStream(roomId: string): DomainEvent[] {
      return this.events
        .filter(e => e.aggregateId === roomId)
        .sort((a, b) => a.version - b.version);
    }
  }
Un event écrit n'est jamais modifié.
Une erreur ? On émet un event compensatoire.
En prod, ce tableau devient…
  • PostgreSQL — table append-only, trigger qui bloque UPDATE/DELETE
  • EventStoreDB — conçu pour ça, streams natifs, projections intégrées
  • Kafka — event streaming distribué, retention configurable, replay natif
  • DynamoDB — streams + GSI pour les projections cross-aggregate
LE CODE — CE QU'IL FAUT POINTER
La ligne clé : this.events.push(event). C'est tout. Pas de UPDATE, pas de REPLACE, pas de upsert. On ajoute à la fin. Toujours.
Le tableau privé est volontairement simplifié pour la démo — mais l'interface publique (append / getStream) est exactement celle d'un vrai event store. Le contrat est là : on ne lit que par getStream(), on n'écrit que par append(). Rien d'autre ne touche au tableau.
LE LOG CONSOLE — POINTER LA RESSEMBLANCE AVEC LE JEU
Le format n'est pas décoratif. C'est exactement le même format que l'event log dans la démo jeu.
L'EVENT COMPENSATOIRE
"Une erreur ? On n'efface pas. On émet un event compensatoire."
Exemple concret : Alice s'est checkée dans la mauvaise salle → on émet AttendeeCheckedOut pour annuler, puis AttendeeCheckedIn dans la bonne salle. Le log garde la trace de l'erreur ET de la correction. C'est une feature, pas un bug. En CRUD on aurait juste fait un UPDATE silencieux et perdu l'information que l'erreur a eu lieu.
Analogie jeu : dans un replay, si le joueur fait un mauvais mouvement, le fichier replay l'enregistre quand même. L'histoire est complète.
LA CARD "EN PROD"
"Ouais mais un tableau en mémoire c'est pas sérieux".
"Évidemment, en prod ce tableau devient une vraie persistence. Les options sont nombreuses — PostgreSQL suffit pour commencer, EventStoreDB si vous voulez les streams natifs, Kafka si vous êtes déjà dans un contexte event-driven."

Les projections — replay pur

// projections.ts — reconstruction par replay
  // 🎮 projectRoom() ≡ gameLoop.replayInputs()
  export function projectRoom(
    roomId: string,
    events: DomainEvent[]  // le log complet
  ): RoomState | null {
    if (events.length === 0) return null;

    // État vide initial — comme l'état de départ du jeu
    let state: RoomState = {
      roomId, attendees: new Map(),
      isFull: false, isOpen: false, version: 0,
      eventsReplayed: 0,
    };

    // On rejoue chaque event dans l'ordre
    // Exactement comme un moteur de jeu rejoue les inputs
    for (const event of events) {
      state = applyEvent(state, event); // ← fonction pure
      state.eventsReplayed++;
    }
    return state; // état reconstruit
  }
// applyEvent = applyInput dans le jeu
  // (état + event) → nouvel état, sans effet de bord
  function applyEvent(state, event): RoomState {
    switch (event.type) {
      case "AttendeeCheckedIn": {
        const attendees = new Map(state.attendees);
        attendees.set(event.payload.attendeeId, event.payload.attendeeName);
        return { ...state, attendees, isFull: attendees.size >= state.capacity };
      }
      // ...
    }
  }

Le flux complet
1.Commande reçue (HTTP POST)
2.Charger le stream
3.Projeter l'état courant
4.Valider les règles métier
5.Émettre 0..N events

RoomFull est émis automatiquement par le handler.
Le client ne décide pas. La règle vit ici. 
export function checkIn(
  roomId: string, attendeeId: string, attendeeName: string
): void {

  const events = eventStore.getStream(roomId);

  const state  = projectRoom(roomId, events);

  if (!state || !state.isOpen)
    throw new Error("Room not found or closed");
  if (state.isFull)
    throw new Error("Room is full");

  const toAppend: DomainEvent[] = [{
    type: "AttendeeCheckedIn",
    version: state.version + 1,
    payload: { attendeeId, attendeeName },
  }];
  if (state.attendees.size + 1 >= state.capacity) {
    toAppend.push({ type: "RoomFull", version: state.version + 2, payload: {} });
  }

  eventStore.append(toAppend);
}
ÉTAPE 1 — COMMANDE REÇUE L'API reçoit un POST /rooms/amphi-a/checkin avec { attendeeId, attendeeName }. C'est une intention, pas un fait. On ne sait pas encore si elle va réussir. Analogie jeu : le joueur appuie sur une touche. Le moteur n'a pas encore décidé ce qui se passe.
ÉTAPE 2 — CHARGER LE STREAM eventStore.getStream(roomId) retourne tous les events de cette salle dans l'ordre. Pas de SELECT * FROM rooms WHERE id=... — on charge l'historique complet. En prod avec snapshots : on charge le snapshot + les events postérieurs seulement.
ÉTAPE 3 — PROJETER L'ÉTAT COURANT projectRoom() rejoue les events un par un pour reconstruire l'état actuel. Fonction pure — même events → même état, toujours. C'est ici qu'on sait combien de gens sont dans la salle en ce moment.
ÉTAPE 4 — VALIDER LES RÈGLES MÉTIER Trois vérifications : la salle existe et est ouverte, elle n'est pas pleine, le participant n'est pas déjà inscrit. Si une règle échoue, on lève une erreur — aucun event n'est émis. Rien ne s'est passé dans le store.
ÉTAPE 5 — ÉMETTRE LES EVENTS (le cœur du talk) C'est ici que RoomFull entre en jeu. Si le check-in porte le nombre de participants à la capacité max, le handler émet RoomFull automatiquement — sans que le client l'ait demandé. Le client a fait une action (checkIn), le moteur métier a déduit la conséquence (RoomFull). Exactement comme le joueur appuie sur SHOOT et le moteur décide PLAYER_HIT si la hitbox correspond.
ÉTAPE 6 — APPEND ONLY eventStore.append() écrit les events dans le log. Jamais de UPDATE, jamais de DELETE. Une erreur future sera un event compensatoire, pas une correction en base. Une fois append() appelé, c'est un fait historique.

Les snapshots — save states

// snapshots.ts — optimisation de la relecture
  // 🎮 Analogie : les save states d'un émulateur

  class SnapshotStore {
    save(roomId: string, state: RoomState): void {
      this.snapshots.set(roomId, {
        state,
        takenAt:      new Date().toISOString(),
        eventVersion: state.version, // dernier event inclus
      });
      // [💾] SNAPSHOT  { roomId, savedAtVersion: 6 }  ← save state
    }

    // Prendre un snapshot tous les N events
    shouldSnapshot(
      eventsReplayed: number,
      threshold = 50    // ← configurable
    ): boolean {
      return eventsReplayed >= threshold;
    }
  }

  // Usage : au lieu de rejouer depuis le début…
  // 1. Charger le dernier snapshot (état à version N)
  // 2. Rejouer uniquement les events après N

Un émulateur peut sauvegarder l'état complet de la machine à un instant T.

Limites — soyons honnêtes

⚡
Coût de relecture — O(n) Sans snapshot, chaque requête rejoue tous les events. Sur des streams de milliers d'events, ça coûte. Mitigation : snapshots périodiques, caches de projection.
🔧
Pas de correction directe Une erreur = un event compensatoire. Pas de UPDATE en base. Mental model différent pour les équipes habituées au CRUD.
📋
Versioning des events Si le schéma d'un event change, les anciens snapshots deviennent incompatibles. Il faut versionner les events et gérer les migrations de schéma.
🧰
Complexité outillage Migrations, debugging plus indirect, tests d'intégration plus complexes. Le tooling event sourcing est moins mature que le tooling SQL classique.
📊
Requêtes cross-aggregates coûteuses "Toutes les salles avec plus de 10 check-ins ce mois-ci" = rejouer tous les streams. Les projections doivent être construites et maintenues à part (read models).

  • Audit réglementaire obligatoire — finance, santé, légal
  • Workflow complexe — commandes, réservations, processus métier long
  • Debug en prod critique — "on doit savoir exactement ce qui s'est passé"
  • Systèmes distribués — Kafka, CQRS, microservices event-driven
  • Collaboration temps réel — Google Docs = event sourcing
Règle pragmatique : si votre client ne vous demande jamais "qu'est-ce qui s'est passé ici ?", vous n'avez probablement pas besoin d'event sourcing.

LES MAUVAIS CANDIDATS — LES DÉFENDRE FRANCHEMENT
CRUD simple : "Un catalogue produits. Un CMS. Une table de configuration. Des settings utilisateur. Vous avez besoin de lire et d'écrire un état courant. L'historique n'apporte aucune valeur métier. Restez en CRUD. Vraiment. C'est le bon outil pour ça, et n'importe quel dev de votre équipe peut le maintenir sans formation spécifique."
Équipe non formée : "C'est probablement le critère le plus important en pratique. L'event sourcing mal compris, c'est pire que du CRUD mal fait. Vous vous retrouvez avec un système qui a la complexité de l'event sourcing sans aucun de ses bénéfices — parce que personne ne sait écrire des projections correctes, personne ne comprend l'immuabilité, et les events finissent par être des déguisements de tables CRUD."
"C'est ce qu'on appelle du cargo cult. Vous avez les formes sans le fond. C'est le pire des deux mondes."
Deadline serrée : "La courbe d'apprentissage est réelle. Comptez plusieurs semaines avant qu'une équipe soit à l'aise. Sur un sprint de deux semaines pour livrer un MVP, ce n'est pas le moment d'introduire ce pattern. Livrez d'abord, refactorez ensuite si les besoins le justifient."
LES BONS CANDIDATS — ÊTRE PRÉCIS
Audit réglementaire : "Finance, santé, légal — partout où vous devez prouver ce qui s'est passé à un régulateur. L'audit trail gratuit n'est plus un nice-to-have, c'est une obligation. Et le construire par-dessus du CRUD coûte cher et reste fragile."
Workflow complexe : "Une commande e-commerce : créée, validée, en préparation, expédiée, livrée, retournée. Chaque transition est un event métier. L'historique a de la valeur — le client veut savoir où est son colis, le support veut savoir ce qui s'est passé, la finance veut tracer chaque changement de statut."
Google Docs : "Google Docs est de l'event sourcing. Chaque frappe, chaque modification est un event. C'est ce qui permet l'historique des versions, la collaboration temps réel, le undo/redo illimité. Quand vous avez ce type de besoin — collaboration, versioning, undo — event sourcing s'impose."
LA RÈGLE PRAGMATIQUE — LA DIRE À VOIX HAUTE

"Si votre client ne vous demande jamais 'qu'est-ce qui s'est passé ici ?' vous n'avez probablement pas besoin d'event sourcing."

"C'est le test le plus simple. Posez-vous cette question avant de commencer. Si la réponse est 'jamais' ou 'rarement', gardez le CRUD. Si la réponse est 'tout le temps' ou 'c'est critique pour nous' — vous avez votre réponse."

Ce qu'on a vu ensemble

01
Un replay de jeu = état initial + log d'events rejoués par une fonction pure. Pas une vidéo. Jamais.
02
L'event sourcing applique exactement cette idée à nos SI : on stocke les décisions métier, pas l'état final.
03
La projection est une fonction pure : f(events) → état. Déterministe = time travel possible.
04
Use cases réels : audit trail gratuit, debug en prod, migration de schéma, reconstruction sur nouvelle archi.
05
Mais : coût de relecture, versioning, complexité outillage. NE PAS utiliser pour du CRUD simple.
