Le killcam — tout le monde connaît

Vous mourez dans Counter-Strike, Overwatch, Fortnite…

Le jeu vous montre les 5 dernières secondes depuis le point de vue de votre adversaire.

Question : le jeu a-t-il enregistré une vidéo de ces 5 secondes en permanence ?

Non. Jamais.

Une vidéo 60fps 1080p = ~150 Mo/min.
Une session de jeu de 30 min = 4,5 Go.
En permanence, en RAM. Non.

Ce que le jeu stocke vraiment
[t=0] GAME_START seed=42
[t=1] MOVE_RIGHT p=1
[t=4] JUMP p=1
[t=9] SHOOT p=2, dir=W
[t=12] PLAYER_HIT ← auto

Quelques dizaines d'octets par frame d'input

Ici, la réponse doit tomber sèchement. « Non. Jamais. » Puis tu enchaînes : si c'était une vraie vidéo 1080p 60 fps stockée en continu, ce serait monstrueux en bande passante, en mémoire, en stockage. Donc non, ce n'est pas ça que le jeu garde. Joue bien l'effet de rupture : c'est le premier petit moment “aha”. La révélation, c'est qu'on ne stocke pas des images. On stocke l'histoire de la partie.

La formule du replay

État initial + Log d'events = Replay identique
État initial

Position de départ, graine du générateur aléatoire (RNG seed), configuration de la map, stats des joueurs.

→ Reproductible à l'identique

Log d'events

Uniquement les changements d'inputs (pas 60 snapshots/sec), plus les événements système (mort, victoire).

→ Append-only, compact

Résultat

En réinjectant les events dans le moteur physique depuis l'état initial, on reconstruit exactement la même partie.

→ Déterministe

Le moteur physique est une fonction pure : f(état, input) → nouvel_état
C'est la slide la plus importante de cette première partie, donc ralentis. Quand tu montres la formule, dis-la simplement : « Si je prends un état, que j'applique un input, j'obtiens un nouvel état. Et surtout : même entrée, même sortie. » Le mot clé, c'est déterminisme. Ou au moins quasi-déterminisme maîtrisé. Tu peux ajouter : « Si ma physique regarde l'heure système, le réseau, ou fait n'importe quoi avec le hasard, mon replay casse. » Cette slide prépare directement applyEvent côté backend.

⚠️ La limite : le non-déterminisme

Un bug de replay arrive quand un élément externe perturbe la reconstruction : latence réseau non rejouée, timer système, accès fichier… Exactement les mêmes problèmes qu'en event sourcing.

Prends les exemples comme des preuves historiques, pas comme un détour nostalgie. « Doom le faisait déjà. Les .lmp, ce n'était pas des vidéos : c'était une partition d'inputs. » Puis sur Counter-Strike 2 : « Le killcam n'existe pas comme objet autonome. C'est une reconstruction. La caméra est une illusion. Le vrai matériau, c'est le log. » L'idée que tu veux faire passer : cette approche n'est pas exotique, elle est éprouvée parce qu'elle est compacte et robuste.

CRUD vs Event Sourcing

🗃️ Approche CRUD classique

On stocke l'état courant. Chaque UPDATE écrase le précédent.

  -- amphi-a | 2         | open

  -- Ce qu'on a PERDU :

  -- Pourquoi la salle était pleine à 14h30 ?
  -- Qui a fait le checkout ?
État final uniquement · Passé effacé

On stocke la chronologie des décisions. L'état se reconstruit par replay.

C'est la slide de rupture conceptuelle, donc prends ton temps. « En CRUD, on photographie l'état courant. Les domain events — immuables, nommés au passé

Règles des events
  • Nommés au passé
    RoomOpened ≠ OpenRoom
  • Immuables — jamais modifiés
  • Append-only — jamais supprimés
  • Portent leur payload complet
  • Horodatés — occurredAt
  • Versionnés dans le stream
// events.ts — types immuables, nommés au passé
  export type DomainEvent =
    | RoomOpened        // ← passé, pas "OpenRoom"
    | AttendeeCheckedIn
    | AttendeeCheckedOut
    | RoomFull          // ← émis AUTO par la règle métier
    | RoomClosed;

  interface BaseEvent {
    eventId:     string;   // identifiant unique
    aggregateId: string;   // roomId — "amphi-a"
    occurredAt:  string;   // ISO 8601 — quand ?
    version:     number;   // position dans le stream
  }

  export interface AttendeeCheckedIn extends BaseEvent {
    type: "AttendeeCheckedIn";
    payload: {
      attendeeId:   string; // qui ?
      attendeeName: string;
    };
  }
Sur les événements, insiste sur le vocabulaire. « On parle au passé parce qu'on décrit des faits. Pas des intentions. Pas des ordres. Des faits. » Tu peux faire ressortir trois choses : qui, quand, sur quel aggregate. Et le détail qui vaut de l'or : RoomFull n'est pas quelque chose que le client décide. C'est une conséquence métier émise automatiquement par le domaine. C'est souvent là qu'on sent la différence entre une simple liste de logs et un vrai modèle événementiel.

L'event store — append-only, toujours

// eventStore.ts — le cœur du système
  class EventStore {
    // Le log global
    // En prod : PostgreSQL, EventStoreDB, Kafka…
    private events: DomainEvent[] = [];

    // APPEND ONLY — jamais de UPDATE ni DELETE
    append(events: DomainEvent[]): void {
      for (const event of events) {
        this.events.push(event);

        // Log formaté — le parallèle avec le jeu
        const v    = String(event.version).padStart(3,' ');
        const type = event.type.padEnd(24,' ');
        const data = JSON.stringify(event.payload);
        console.log(`[${v}] ${type} ${data}`);
      }
    }

    // Tous les events d'une salle, dans l'ordre
    getStream(roomId: string): DomainEvent[] {
      return this.events
        .filter(e => e.aggregateId === roomId)
        .sort((a, b) => a.version - b.version);
    }
  }
En production
  • PostgreSQL — table append-only, trigger sur UPDATE/DELETE
  • EventStoreDB — conçu pour ça, streams natifs
  • Kafka — event streaming distribué, retention configurable
  • DynamoDB — streams + GSI pour la projection
Un event écrit n'est jamais modifié.
Une erreur ? On émet un event compensatoire.
Présente l'event store comme l'équivalent direct du fichier replay. « Le cœur du système, ce n'est pas une ligne qui contient la vérité finale. C'est un journal append-only. On ajoute. On n'écrase pas. » Martèle ce point : jamais d'UPDATE, jamais de DELETE sur l'histoire métier. S'il y a une erreur, on émet un autre événement qui la compense ou la corrige. Le log console est utile visuellement : tu veux que les gens voient défiler une chronologie, pas juste une structure de données abstraite.

Les projections — replay pur

// projections.ts — reconstruction par replay
  // 🎮 projectRoom() ≡ gameLoop.replayInputs()

  export function projectRoom(
    roomId: string,
    events: DomainEvent[]  // le log complet
  ): RoomState | null {
    if (events.length === 0) return null;

    // État vide initial — comme l'état de départ du jeu
    let state: RoomState = {
      roomId, attendees: new Map(),
      isFull: false, isOpen: false, version: 0,
      eventsReplayed: 0,
    };

    // On rejoue chaque event dans l'ordre
    // Exactement comme un moteur de jeu rejoue les inputs
    for (const event of events) {
      state = applyEvent(state, event); // ← fonction pure
      state.eventsReplayed++;
    }

    return state; // état reconstruit
  }

  // applyEvent = applyInput dans le jeu
  // (état + event) → nouvel état, sans effet de bord
  function applyEvent(state, event): RoomState {
    switch (event.type) {
      case "AttendeeCheckedIn": {
        const attendees = new Map(state.attendees);
        attendees.set(event.payload.attendeeId,
                      event.payload.attendeeName);
        return { ...state, attendees,
                isFull: attendees.size >= state.capacity };
      }
      // ...
    }
  }
Le parallèle exact
🎮 Jeu ⚙️ Backend
replayInputs() ↔ projectRoom()
applyInput() ↔ applyEvent()
GameState ↔ RoomState
tick ↔ version
seed RNG ↔ état initial
Fonction pure = même events → même état, toujours.
C'est ce qui rend le time travel possible.
C'est la slide centrale du talk. SAVOURE-LA !!! « projectRoom(), c'est replayInputs(). applyEvent(), c'est applyInput(). » Dis-le lentement, presque comme une révélation. Le point vraiment important : la projection est une fonction. Pas d'appel réseau, pas de dépendance à l'heure courante, pas d'état global magique. Même log, même résultat. C'est souvent ici que la salle a le déclic complet. Laisse-leur une seconde pour l'avoir.

Command handlers — la règle métier décide

Le flux complet
1.Commande reçue (HTTP POST)
2.Charger le stream
3.Projeter l'état courant
4.Valider les règles métier
5.Émettre 0..N events
6.Append dans le store
RoomFull est émis automatiquement par le handler.
Le client ne décide pas. La règle vit ici. 
// commands.ts — la logique métier
  export function checkIn(
    roomId: string,
    attendeeId: string,
    attendeeName: string
  ): void {
    // 1. Charger + projeter l'état courant
    const events = eventStore.getStream(roomId);
    const state  = projectRoom(roomId, events);

    // 2. Valider les règles métier
    if (!state || !state.isOpen)
      throw new Error("Room not found or closed");
    if (state.isFull)
      throw new Error("Room is full");

    // 3. Construire les events à émettre
    const toAppend: DomainEvent[] = [{
      type: "AttendeeCheckedIn",
      version: state.version + 1,
      payload: { attendeeId, attendeeName },
      // ...
    }];

    // ★ Règle métier : RoomFull émis auto
    // Le CLIENT ne décide pas — le HANDLER décide
    if (state.attendees.size + 1 >= state.capacity) {
      toAppend.push({
        type: "RoomFull",
        version: state.version + 2,
        payload: { capacity: state.capacity },
      });
    }

    // 4. Écrire dans le store (append only)
    eventStore.append(toAppend);
  }
Sur cette slide, raconte une séquence métier complète. « Une commande arrive. On recharge le stream. On projette l'état courant. On vérifie les règles métier. Puis le système décide quels faits doivent être émis. » Insiste sur RoomFull : « Le client ne demande jamais “mets la salle pleine”. Comme dans un jeu, il fait une action. Et le moteur métier déduit les conséquences. » Très bonne analogie : le joueur appuie sur une touche, mais c'est le moteur qui décide qu'il y a collision, hit, mort, score, etc.

// eventStore.ts — time travel
  // 🎮 Analogie : rejouer jusqu'à la frame N

  getStreamAt(roomId: string, until: Date): DomainEvent[] {
    return this.getStream(roomId).filter(
      e => new Date(e.occurredAt) <= until
      //   ↑ on s'arrête à cette date
      //   tout ce qui suit est ignoré
    );
  }

  // api.ts — ?at= active le time travel
  const events = atParam
    ? eventStore.getStreamAt(roomId, new Date(atParam))
    : eventStore.getStream(roomId);  // état courant

  const state = projectRoom(roomId, events);
  // → état reconstruit à cet instant précis

Snapshots · Complexité · Quand NE PAS l'utiliser
Transition importante pour la crédibilité. Dis-le franchement : « Je ne suis pas là pour vous vendre l'event sourcing comme une religion. Je suis surtout là pour vous éviter de l'utiliser au mauvais endroit. » Cette phrase remet tout le monde avec toi.

Les snapshots — save states

// snapshots.ts — optimisation de la relecture
  // 🎮 Analogie : les save states d'un émulateur

  class SnapshotStore {
    save(roomId: string, state: RoomState): void {
      this.snapshots.set(roomId, {
        state,
        takenAt:      new Date().toISOString(),
        eventVersion: state.version, // dernier event inclus
      });
      // [💾] SNAPSHOT  { roomId, savedAtVersion: 6 }  ← save state
    }

    // Prendre un snapshot tous les N events
    shouldSnapshot(
      eventsReplayed: number,
      threshold = 50    // ← configurable
    ): boolean {
      return eventsReplayed >= threshold;
    }
  }

  // Usage : au lieu de rejouer depuis le début…
  // 1. Charger le dernier snapshot (état à version N)
  // 2. Rejouer uniquement les events après N
  // → O(events_depuis_snapshot) au lieu de O(total)
🎮 Save state = snapshot

Un émulateur peut sauvegarder l'état complet de la machine à un instant T.

Pour reprendre, on charge le save state et on continue — pas besoin de rejouer depuis le début.

⚠️ Limite importante

Le snapshot ne remplace pas le log d'events.
Audit trail et time travel avant le snapshot nécessitent toujours le log complet.

Sur les snapshots, reste très pragmatique. « Rejouer 10 000 événements à chaque requête, c'est élégant sur les slides, mais en prod ça finit par coûter. Donc on fait comme les émulateurs avec les save states : on prend un point de reprise. » Puis la nuance essentielle : « Le snapshot accélère la reconstruction, il ne remplace pas le log. » Si tu veux marquer les esprits : « Le snapshot est un accélérateur, pas une substitution. »

Limites — soyons honnêtes

⚡
Coût de relecture — O(n) Sans snapshot, chaque requête rejoue tous les events. Sur des streams de milliers d'events, ça coûte. Mitigation : snapshots périodiques, caches de projection.
🔧
Pas de correction directe Une erreur = un event compensatoire. Pas de UPDATE en base. Mental model différent pour les équipes habituées au CRUD.
📋
Versioning des events Si le schéma d'un event change, les anciens snapshots deviennent incompatibles. Il faut versionner les events et gérer les migrations de schéma.
🧰
Complexité outillage Migrations, debugging plus indirect, tests d'intégration plus complexes. Le tooling event sourcing est moins mature que le tooling SQL classique.
📊
Requêtes cross-aggregates coûteuses "Toutes les salles avec plus de 10 check-ins ce mois-ci" = rejouer tous les streams. Les projections doivent être construites et maintenues à part (read models).
Fais apparaître les limites une à une et commente-les franchement. « Les coûts de relecture sont réels. Le versioning des events est difficile. Les projections et les read models ajoutent de la complexité. Le tooling est moins banal qu'un CRUD classique. Et l'onboarding des équipes est plus exigeant. » Tu veux que la salle sente que tu n'essaies pas de vendre du rêve. C'est souvent cette honnêteté qui rend tout le reste beaucoup plus convaincant.

Quand NE PAS utiliser l'event sourcing

❌ Mauvais candidats
  • CRUD simple — catalogue produits, CMS, settings
  • Faible volume — quelques dizaines d'entités, pas d'audit nécessaire
  • Pas besoin d'historique — "on veut juste stocker l'état courant"
  • Équipe non formée — sans investissement formation, ça devient du cargo cult
  • Deadline serrée — la courbe d'apprentissage est réelle
✅ Bons candidats
  • Audit réglementaire obligatoire — finance, santé, légal
  • Workflow complexe — commandes, réservations, processus métier long
  • Debug en prod critique — "on doit savoir exactement ce qui s'est passé"
  • Systèmes distribués — Kafka, CQRS, microservices event-driven
  • Collaboration temps réel — Google Docs = event sourcing
Règle pragmatique : si votre client ne vous demande jamais "qu'est-ce qui s'est passé ici ?", vous n'avez probablement pas besoin d'event sourcing.
Slide importante : le fameux “quand ne pas l'utiliser”. Dis très clairement : « Si vous avez juste besoin d'éditer un état courant et que l'historique détaillé n'apporte pas de valeur, restez en CRUD. Vraiment. » Citer des contre-exemples simples : CRUD d'administration banal, référentiels simples, cas où l'audit fin n'a pas de valeur métier. L'idée à faire passer : event sourcing est un outil spécialisé pour des problèmes où l'histoire a une valeur. Pas un badge d'architecture cool.

Ce qu'on a vu ensemble

01
Un replay de jeu = état initial + log d'events rejoués par une fonction pure. Pas une vidéo. Jamais.
02
L'event sourcing applique exactement cette idée à nos SI : on stocke les décisions métier, pas l'état final.
03
La projection est une fonction pure : f(events) → état. Déterministe = time travel possible.
04
Use cases réels : audit trail gratuit, debug en prod, migration de schéma, reconstruction sur nouvelle archi.
05
Mais : coût de relecture, versioning, complexité outillage. NE PAS utiliser pour du CRUD simple.
Fais apparaître les takeaways un par un, avec un vrai rythme de fin de conf. À chaque point, reformule oralement plutôt que de lire. Sur le dernier, appuie fort : « Ce n'est pas parce que la métaphore est belle, ni parce que la démo est cool, qu'il faut en mettre partout. » C'est ta dernière grande note de pragmatisme avant la conclusion.
